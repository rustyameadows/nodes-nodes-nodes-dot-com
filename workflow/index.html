Workflow

How the workflow actually moves.

Start in a local project, build a graph with notes, data, files, tools, and model nodes, run jobs through your own providers, then inspect the queue and compare outputs without losing the thread.

Sequence

A simple path through the app.

Project

Open a local project

Create or reopen a workspace so the files, metadata, and outputs stay attached to one place.

Canvas

Build the graph

Connect prompt notes, lists, templates, source files, and model nodes on the infinite canvas.

Run

Run with your own keys

Send jobs through your own provider credentials while the queue makes execution state visible.

Compare

Read the run, then review

Inspect queue state, request-response detail, and outputs before feeding the good results into the next pass.

The canvas is where notes, assets, model nodes, and connections become a real workflow.

The queue is where you inspect whether the run moved and what the provider said back.

Who it is for

People doing real exploratory work

Especially the ones already bouncing between prompts, datasets, assets, provider tabs, and output folders.

Current scope

Mac first. Local first. Single-user.

Right now it is a desktop app, not a shared cloud workspace, and that boundary is part of the product shape.
